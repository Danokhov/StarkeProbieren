# Масштабируемость Firebase Firestore для прогресса пользователей

## Текущая структура данных

### Коллекция: `users`
- **Документ ID**: `telegramId` (строка, уникальный идентификатор пользователя из Telegram)

### Структура документа:
```typescript
{
  telegramId: string;
  name: string;
  progress: Record<string, string[]>,           // topicId -> массив завершенных модулей
  itemProgress?: Record<string, Record<string, Record<number, boolean>>>, // детальный прогресс
  updatedAt: timestamp
}
```

**Пример:**
```json
{
  "telegramId": "123456789",
  "name": "Иван Иванов",
  "progress": {
    "house-cleaning": ["video", "mantras", "exercises"],
    "job-interview": ["video", "mantras"]
  },
  "itemProgress": {
    "house-cleaning": {
      "exercises": {
        "0": true,
        "1": true,
        "2": false,
        "15": true
      }
    }
  },
  "updatedAt": "2024-01-15T10:30:00Z"
}
```

## Оценка размера данных

### Для 1 пользователя с 11 модулями:

**1. `progress` (основной прогресс):**
- 11 тем × ~20 символов (ID темы) = ~220 байт для ключей
- Массивы завершенных модулей: ~11 тем × ~50 символов = ~550 байт
- **Итого: ~0.8 KB**

**2. `itemProgress` (детальный прогресс по упражнениям):**
- Предположим: ~50 упражнений на модуль (mantraGapExercises)
- 11 модулей × 50 упражнений = 550 упражнений
- Каждое упражнение: `"индекс": boolean` = ~10 байт
- **Итого: ~5-10 KB** (в зависимости от количества пройденных упражнений)

**3. Метаданные:**
- `telegramId`, `name`, `updatedAt`: ~100 байт

**Итого на 1 пользователя: ~6-11 KB**

### Для 500 пользователей:

- **Минимально** (только основной прогресс): 500 × 1 KB = **~0.5 MB**
- **Среднее** (с детальным прогрессом ~50% упражнений): 500 × 6 KB = **~3 MB**
- **Максимально** (все упражнения пройдены): 500 × 11 KB = **~5.5 MB**

## Ограничения Firestore

### ✅ Выдержит ли база?

**ДА! База легко выдержит 500 пользователей и 11 модулей:**

1. **Размер коллекции:**
   - 5.5 MB для 500 пользователей - это **очень мало**
   - Firestore может хранить **миллионы документов**

2. **Размер документа:**
   - Максимум: **1 MB на документ**
   - Наш документ: **~6-11 KB** (в 100 раз меньше лимита)

3. **Количество записей:**
   - Firestore может хранить **практически неограниченное** количество документов в коллекции

4. **Производительность:**
   - Чтение одного документа: **~50-100ms**
   - Запись одного документа: **~100-200ms**
   - Подходит для нашего случая использования

## Рекомендации по оптимизации

### ✅ Текущая реализация уже оптимизирована:

1. **Один документ на пользователя:**
   - ✅ Эффективно - одна операция чтения/записи на пользователя
   - ✅ Все данные пользователя в одном месте

2. **Использование `merge: true`:**
   - ✅ Обновляет только измененные поля
   - ✅ Не перезаписывает весь документ

3. **Индексирование по `telegramId`:**
   - ✅ Быстрый поиск по ID документа
   - ✅ O(1) доступ к данным пользователя

### ⚠️ Потенциальные улучшения (если вырастет до 10,000+ пользователей):

1. **Ограничение `itemProgress`:**
   - Если упражнений станет очень много (>1000 на модуль), можно ограничить сохранение только последних N упражнений
   - Или вынести в отдельную подколлекцию

2. **Батчинг записей:**
   - Если много пользователей обновляют прогресс одновременно, использовать батчи (но для 500 пользователей это не критично)

3. **Кэширование:**
   - Уже есть fallback на localStorage
   - Можно добавить кэширование прогресса в памяти на короткое время
